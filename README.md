A TUI framework for building terminal interfaces in Zig.

https://github.com/fern-tui/fern-core


## Documentation

The examples in `examples/` are the primary documentation. Each is a single
self-contained file demonstrating a complete init/update/view cycle.

Every public function in the source carries a doc comment. API documentation
can be generated by running:

```sh
zig build docs
```

## Requirements

Zig 0.16.0. Not 0.14, not 0.13, not a nightly build.
Get it at https://ziglang.org/download.

## Running the Examples

Clone the repository and run one of the three included examples:

```sh
git clone https://github.com/fern-tui/fern-core.git
cd fern-core

zig build example-spinner    # a spinner ticking at 60fps, press q to quit
zig build example-progress   # a spring-animated progress bar with RGB gradient
zig build example-list       # a paginated list with keyboard navigation
```

Source for each example lives in `examples/`. Read it before asking questions.

## Adding fern to Your Project
> No release has been published yet. This section documents the intended workflow.

Add the dependency to your `build.zig.zon`:

```zig
.dependencies = .{
    .fern = .{
        .url = "https://github.com/fern-tui/fern-core/archive/refs/tags/v0.1.5-beta9.tar.gz",
        .hash = "...",
    },
},
```

Fetch and lock the hash:

```sh
zig fetch --save https://github.com/fern-tui/fern-core/archive/refs/tags/v0.1.5-beta9.tar.gz
```

Wire it up in `build.zig`:

```zig
const fern = b.dependency("fern", .{
    .target = target,
    .optimize = optimize,
});

exe.root_module.addImport("fern_app",    fern.module("fern_app"));
exe.root_module.addImport("fern_widget", fern.module("fern_widget"));
exe.root_module.addImport("fern_style",  fern.module("fern_style"));
exe.root_module.addImport("fern_ansi",   fern.module("fern_ansi"));
```

Import only the modules your application uses.

## Architecture

fern is not a monolith. It is six composable libraries that happen to fit
together cleanly. Each can be used independently. Nothing reaches up the
dependency graph. No circular imports.

The application model is the Elm Architecture: init, update, view. Every
frame is a pure function of your state. Side effects are values you return
through the `Cmd` type. The runtime executes them. No hidden threads you did
not ask for, no allocations behind your back.

The six modules and their dependency rules:

| Module      | May import                                  |
|-------------|---------------------------------------------|
| fern_ansi   | nothing -- std only                         |
| fern_anim   | nothing -- std.math only                    |
| fern_style  | fern_ansi                                   |
| fern_zone   | fern_ansi                                   |
| fern_app    | fern_ansi, fern_anim, fern_zone             |
| fern_widget | fern_style, fern_anim, fern_zone, fern_app  |

This table is enforced in code review, not just documented. A PR that
introduces a dependency cycle is closed regardless of the feature it delivers.

`fern_ansi` parses every byte the terminal sends. `fern_anim` provides damped
spring and projectile physics. `fern_style` implements chainable style
building with proper `inherit()` semantics. `fern_zone` tracks where widgets
land on screen and handles mouse hit-testing. `fern_app` is the runtime: raw
mode, SIGWINCH, a 60fps event loop, and four worker threads. `fern_widget`
contains spinner, progress bar, viewport, text input, list, paginator, and
more.

The diff renderer performs line-level diffing at 60fps and repaints only what
changed. The zone manager embeds invisible private-use CSI escape sequences as
coordinate markers in render output, which maps every mouse event to the exact
widget that owns that cell.

fern targets DO-178C compliance, the safety standard for airborne software.
This shapes every decision about allocations, error handling, and test
coverage.

## Testing

```sh
zig build test          # all modules
zig build test-app      # app/ only
zig build test-widget   # widget/ only
```

Run `zig build test` before opening a pull request. If tests fail before your
changes, open an issue rather than a PR.

## Contributing

fern is free and open-source software. Bug reports and patches are welcome
from everyone. fern governance is BDFN (Benevolent Dictator For Now), which
means the fern-tui team has final say on the design and implementation of
everything.

### Make Software With fern

Using fern in an open-source project is one of the best contributions you can
make. It surfaces real bugs with real motivations, which makes proposals and
feature requests straightforward to reason about. A project built with fern
also helps develop skills and builds a portfolio.

### Talk About fern

Writing about fern, or speaking about it at a conference, is a meaningful
contribution. Libraries grow through their ecosystems. More people building
with fern means more real-world use cases, more edge cases discovered, and
more improvements for everyone.

### AI Usage Policy

fern does not adopt Zig's no-LLM policy. AI tools are welcome; shipping
code you cannot explain is not.

AI tools may be used to assist development. AI was used to generate doc
comments and documentation drafts in this repository. The code, architecture,
and design are handwritten Zig 0.16.0.

The rule for contributions is simple: you are responsible for every line you
submit. The following will get a PR closed:

- You cannot explain a line in code review. "The model wrote it" is not a
  review response. It means you submitted code you do not understand into a
  codebase other people depend on.

- The code uses Zig APIs from a version prior to 0.16.0. Most language models
  were trained before 0.16.0 and will confidently produce code for older
  versions. Allocator interfaces changed. Several things that compiled in 0.14
  do not compile in 0.16.0. Verify every API call against the 0.16.0 stdlib
  source. The source is the ground truth. The model is not.

- Heap allocations were introduced into hot render paths.

- The diff is a wall of changes with no coherent description. If you cannot
  describe what the PR does and why in one paragraph, it is not ready.

If a bug ships because you did not verify what the model generated, that bug
has your name on it. You reviewed it. You submitted it. You own it.

### Find a Contributor Friendly Issue

Look for issues tagged `good first issue` or `help wanted` before starting
new work. For new ideas, open an issue first. A PR nobody asked for is a PR
that will wait indefinitely.

Proposals still under consideration carry risk: an implementation may not be
accepted. If you are interested in an open proposal, express that interest in
the issue tracker with additional context or real-world use cases. A concrete
use case is the strongest argument.

### Editing Source Code

Format the source before every commit, not sometimes:

```sh
zig fmt src/
zig fmt examples/
zig fmt build.zig
```

PRs that skip formatting will be asked to reformat before merge.

See [CONTRIBUTING.md](docs/CONTRIBUTING.md) for the complete picture: coding
style, commit format, how to add a new widget, and what DO-178C targeting
means in practice.
